## 三、哪些功能很好用？

### 1. AI 变化摘要 — 杀手级功能 ⭐⭐⭐⭐⭐

这是我见过的最实用的 diff 展示方式。以 Hacker News 为例，摘要不是简单列 diff，而是结构化地指出：
- 排名变化（"Stop Flock 从第4位跃升至第1位"）
- 新增/删除的具体内容
- 数据趋势（"Claude Code Routines 从 390 增至 409 点"）

**这比 changedetection.io 的 raw diff 强太多了。** changedetection.io 给你看一堆红绿 diff 行，信息密度很低。盯梢的 AI 摘要直接告诉你"发生了什么"，而不是"哪些字符变了"。

### 2. API 设计简洁

```bash
POST /api/monitors          # 添加
POST /api/monitors/{id}/check   # 手动触发
GET  /api/monitors/{id}/changes  # 查看历史
DELETE /api/monitors/{id}       # 删除
```

很直觉，没有过度设计。作为 PM 我自己用 curl 就能操作，不需要开发帮忙。

### 3. 频率可配置

支持按分钟设置检查频率（我设了竞品官网 1440 分钟/天，行业媒体 60 分钟），这符合实际需求——竞品定价页不需要每小时查一次，但新闻频道需要更频繁。

## 四、哪些功能缺失让我很痛苦？

### 🔴 P0: SPA/JS 渲染页面无法抓取

**这是最致命的问题。** 飞书定价页（feishu.cn/pricing）报"内容为空或太短"，说明抓取器拿到的是空壳 HTML。2026 年了，竞品官网 90% 是 SPA，如果只能抓静态 HTML，那大量核心场景直接废了。

**我需要监控的页面：**
- 飞书/钉钉/企微的定价页 → 全是 SPA
- 竞品产品更新日志 → 多数是 SPA
- 招聘网站（Boss直聘/拉勾）→ 重度 JS 渲染

如果不解决这个问题，我最多只能用它监控 36kr、HN 这类静态/SSR 页面，覆盖不到我核心场景的 60%。

**建议：** 提供 Headless Chrome 渲染模式，哪怕只是个开关（`"render": true`），可以接受更长的抓取时间。

### 🔴 P0: 没有通知推送

变化检测到了，但 `notified` 全是 0，设置页返回空 `{}`。意味着：
- 没有 Webhook 推送
- 没有邮件通知
- 没有飞书/钉钉/Slack 集成

**一个监控工具如果不能主动通知我，那我还是得手动来看，只是换了个地方手动刷而已。** 这严重降低了工具的价值——从"自动化监控"退化成"手动查询面板"。

**建议：** 至少支持 Webhook，让我能接到飞书机器人里。

### 🟡 P1: 没有 CSS Selector / XPath 精确选取

我加了 `selector: null`，但即使有这个字段，也没看到实际生效的例子。竞品官网内容很多，我只关心定价表格那一块。不能精确选取意味着：
- 每次变化包含大量噪音（导航栏改个字就触发）
- AI 摘要的注意力被分散到不重要的变化上

### 🟡 P1: 缺少分组/标签管理

我加了 3 个竞品监控、1 个行业媒体，但它们平铺在一个列表里。实际使用中我会有：
- 竞品监控组（飞书/钉钉/企微）
- 行业媒体组（36kr/虎嗅/InfoQ）
- 招聘动向组（Boss直聘搜"飞书"/"钉钉"）

没有分组，监控项超过 10 个就混乱了。

### 🟡 P2: 错误处理不友好

- 飞书定价页报"抓取内容为空或太短"——没告诉我为什么为空，是被反爬了？是 JS 渲染问题？是超时了？
- 36kr 子频道报"HTTP 404"——但用浏览器能正常访问，可能是 User-Agent 或重定向问题
- 没有重试机制，一次失败就变 error 状态

### 🟢 P3: 缺少变化 diff 原文对比

AI 摘要很好，但有时候我需要看精确的原文 diff——比如竞品定价从 ¥399 改成 ¥499，我需要确切数字而不只是"定价有变化"。目前 API 只返回 `ai_summary`，没有 `old_content` / `new_content` 或 diff 字段。

| 维度 | 手动刷网页 | Chrome 插件 (Distill.io) | changedetection.io | 盯梢 |
|------|----------|------------------------|-------------------|------|
| 上手成本 | 零 | 低 | 中（需部署） | 低 |
| SPA 支持 | ✅（人肉） | ✅（浏览器环境） | ✅（Playwright） | ❌ |
| AI 摘要 | ❌ | ❌ | ❌ | ✅⭐ |
| 变化通知 | ❌ | ✅（浏览器通知） | ✅（多渠道） | ❌ |
| 精确选取 | ✅ | ✅（可视化选取） | ✅（CSS/XPath） | ❌ |
| 部署方式 | N/A | 浏览器插件 | Docker | 本地服务 |
| 噪音过滤 | 人工 | 阈值 | 规则 | AI（理论上更好） |
| 协作 | ❌ | ❌ | ✅ | ❌ |

**结论：** 盯梢的 AI 摘要是唯一亮点，但在基础能力（SPA 抓取 + 通知）上跟 changedetection.io 还有明显差距。如果只是抓静态页面，changedetection.io 功能更全；如果需要 AI 摘要，现在的方案是 changedetection.io + 自己写 Webhook 调 LLM。

## 八、总结建议

### 优先做（决定产品生死的）：
1. **JS 渲染模式** — 没有这个，丢掉 60% 的使用场景
2. **Webhook 通知** — 没有这个，从"自动化监控"退化为"查询面板"

### 其次做（决定付费转化的）：
3. **CSS Selector 精确选取** — 减少噪音，提高 AI 摘要质量
4. **监控分组/标签** — 管理超过 10 个监控项时必需
5. **错误诊断信息** — 告诉用户为什么抓取失败、怎么解决

### 锦上添花的：
6. 变化对比原文（diff view）
7. 变化趋势图表
8. 团队协作/共享监控
9. 预设模板（一键添加"监控XX竞品定价页"）

---

**一句话总结：AI 摘要是真正的差异化优势，但地基（抓取能力 + 通知）还不够扎实。先把地基打好，这个产品就有商业化的可能。**
